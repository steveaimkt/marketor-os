# pm-sprint-plan · 스프린트 계획

기능·할일을 무작정 쌓지 않고 **하나의 스프린트 골 아래 팀이 실제로 소화할 만큼만** 커밋하게 만드는 스킬. 용량을 넘긴 계획은 자동으로 무너지므로, 가용 용량을 먼저 산정하고 우선순위대로 담다가 용량에 닿으면 멈춘다. 의존성과 리스크를 미리 못박아 착수 후 놀라지 않게 한다.

## 언제 호출하나

- "스프린트 짜줘"·"이번 주(2주) 스프린트 계획"·"작업 구간 계획"·"이번 스프린트 뭐 담을까"
- 파트너 TFT·프로젝트가 정해진 기간(1~2주) 안에 무엇을 끝낼지 커밋해야 할 때
- 백로그·할일이 쌓였는데 이번 구간에 담을 범위를 용량 기준으로 잘라야 할 때
- 우선순위(pm-prioritization)·로드맵(pm-outcome-roadmap)이 정해진 뒤, 그 상위 계획을 실행 구간으로 내릴 때

경계: 목표를 성과 숫자로 설계하는 일은 pm-okrs, 여러 항목의 순서·근거를 성과로 정렬하는 일은 pm-outcome-roadmap이다. 이 스킬은 확정된 우선순위를 **정해진 기간의 실행 커밋**으로 바꾼다.

## 입력

착수 전 아래를 확인한다. gbrain·프로필에 있으면 다시 묻지 않는다.

```yaml
sprint:
  goal_seed:            # 선택 · 이번 구간에 노리는 한 가지 (없으면 백로그 상위에서 도출)
  duration: "2주"       # 기본 2주 (1주도 가능)
team:
  members: 3            # 참여 인원 (예: 파트너 TFT 3인)
  availability:         # 각자 가용 저해 요인
    - "steve 화·목 출강으로 반나절"
    - "디자이너 공휴일 1일"
  velocity_hint:        # 선택 · 최근 스프린트 실제 소화량 (스토리포인트 or 건수)
backlog:                # 필수 · 우선순위 정렬된 후보 (pm-prioritization 산출물 우선)
  - title: "커뮤니티 온보딩 자동화"
    estimate: 5         # 스토리포인트 or 소요(맨데이)
    ready: true         # 착수 준비 완료 여부(수용기준·추정·블로커 없음)
    depends_on: []
external_deps:          # 선택 · 외부 팀·파트너 의존
  - "결제 연동은 PG사 승인 대기"
```

착수 시 먼저 `mcp__gbrain__recall` 또는 `search`로 태그 `sprint`를 회수해 직전 스프린트의 커밋 대비 완료량(velocity)·미완 이월 항목을 읽는다 (read-back). 실제 소화 속도를 모르면 용량을 과대 산정한다.

## 워크플로 (스프린트 계획)

같은 입력은 같은 구조를 낸다.

1. **용량 산정** · 인원 × 구간일수에서 가용 저해(출강·휴가·회의·상시 대응)를 빼고, 직전 velocity를 반영해 실제 소화 가능량을 낸다. 그중 **15~20%는 예상 못한 일·유지보수용 버퍼**로 떼어 놓는다. (커밋 가능 용량 = 총용량 − 버퍼)
2. **스프린트 골 확정** · 이번 구간이 끝났을 때 무엇이 달라져 있어야 하는지를 **1문장 가치 진술**로 못박는다. 백로그 상위 항목들이 이 골을 향하도록 정렬한다.
3. **스토리 선택·순서** · 우선순위 백로그 상단부터, 각 항목의 **준비 완료(ready) 기준**(명확한 수용기준·추정 완료·블로커 없음)을 확인하며 담는다. 준비 안 된 항목은 담기 전 다듬기(refinement)로 플래그하고 건너뛴다. **커밋 가능 용량에 닿으면 멈춘다.**
4. **의존성 매핑** · 항목 간 선후 관계를 잇고, 의존 작업을 앞 순서로 배치한다. 외부 팀·파트너 의존은 담당(누가 언제까지)을 명시하고, 완료까지 이어지는 **크리티컬 패스**를 표시한다.
5. **리스크 분석** · 복잡도·불확실성 높은 항목, 지연 위험 있는 외부 의존, 특정 1인에게 몰린 지식(knowledge silo)을 찾아 각각 완화책을 붙인다.
6. **gbrain put + 로컬 저장** · `mcp__gbrain__put_page`로 확정 스프린트를 태그 `sprint`로 적재(골·용량·커밋 목록·리스크). 로컬 파일 저장 후, 노션 프로젝트관리 DB 미러링은 승인 게이트.

### ⚠️ 안티패턴 (반드시 피한다)

- **용량 초과 커밋** · 버퍼 없이 100% 채우면 반드시 이월된다. 15~20% 버퍼는 협상 대상이 아니다.
- **준비 안 된 스토리 담기** · 수용기준·추정·블로커 정리 안 된 항목을 담으면 스프린트 중간에 멈춘다. 담기 전 refinement.
- **골 없는 스토리 나열** · 하나의 스프린트 골 없이 잡다한 할일만 모으면 완료 여부를 판정할 수 없다.
- **의존성 무시** · 선행 작업을 뒤로 두면 크리티컬 패스가 막혀 전부 지연된다.

## 산출물

로컬: `outputs/{YYYY-MM-DD}/pm-sprint-plan/{YYYY-MM-DD}-sprint.md`

형식:

```markdown
# {제목} 스프린트 · {날짜}
스프린트 골: 파트너 온보딩 첫 화면을 붙여 신규 파트너가 자기 힘으로 1일차를 완주하게 한다
기간: 2주 (7/19~8/1)
팀 용량: 24pt (3인 × 10 − 출강·휴가 6)
커밋: 20pt / 4건 · 버퍼: 4pt (17%)

## 커밋 스토리
1. 파트너 온보딩 자동화 — 5pt — steve — (선행 없음)
2. 온보딩 안내 뉴스레터 연결 — 3pt — 콘텐츠 — depends: 1
3. 활성도 대시보드 초안 — 8pt — 데이터 — 외부: GA4 권한 대기
4. 1주차 회고 준비 — 4pt — steve — depends: 1,2

## 리스크
- 3번 GA4 권한 지연 위험 → 대체로 시트 수기 집계 폴백, 권한 요청 7/19 발송
- steve 화·목 출강 몰림 → 코어 작업은 월·수·금에 배치

## 크리티컬 패스
1 → 2 → 4 (온보딩이 뉴스레터·회고의 선행)

## 이월·보류
{준비 미완으로 다음 스프린트로 넘긴 항목과 사유}
```

검토 요약(무엇을·왜·한계: 용량 가정·미확정 외부 의존)을 3줄로 붙이고 "이 커밋 범위로 갈까요? 뺄 항목은?" 피드백을 청한다. 구간 종료 시 pm-retro(회고)로 잇고, 상위 계획은 pm-outcome-roadmap·pm-okrs와 연결한다.
