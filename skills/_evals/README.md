# 스킬 검증 규약 (evals)

> 출처: `coreyhaines31/marketingskills` 구조 차용 (2026-07-20 · 45/47 스킬이 보유)
> 목적: **"작동한다"를 말이 아니라 테스트로 증명한다.**

## 왜 필요한가

스킬 100개를 책으로 팔고(『마케터를 위한 클로드 100 Skills』) 유튜브로 무료 배포한다. 그런데 지금은:

- 책: "스킬 100개" · **품질 보증 근거 없음**
- 영상: "그대로 드립니다" · **작동 보장 없음**
- 배포 후: "안 되는데요" 댓글 · **재현 확인 수단 없음**

evals가 있으면 **"검증된 100개"** 가 된다. 이건 경쟁 우위이자 책·영상의 신뢰 자산이다.

## 파일 위치

```
skills/{스킬명}/evals/evals.json     ← marketing-os 자체 스킬
```

⚠️ **MSK 정본 스킬(`msk/plugins/`)은 수정하지 않는다.** MSK 쪽 evals는 MSK 저장소에서 별도 관리한다 (README 드리프트 룰).

## 형식

```json
{
  "skill_name": "스킬명",
  "evals": [
    {
      "id": 1,
      "prompt": "사용자가 실제로 할 법한 요청 (구체적 숫자·상황 포함)",
      "expected_output": "이 요청에 어떤 산출물이 나와야 하는지 서술",
      "assertions": [
        "검증 가능한 조건 1",
        "검증 가능한 조건 2"
      ],
      "files": []
    }
  ]
}
```

## 작성 규칙

1. **prompt는 실제 발화처럼.** "키워드 분석해줘"가 아니라 "화장품 신제품 런칭하는데 네이버 키워드 뭐부터 봐야 해?"
2. **assertions는 이분법으로.** "좋은 결과" ❌ · "TOP 20 표에 검색량·경쟁도 열이 있다" ✅
3. **실패 조건도 넣는다.** "근거 없는 수치를 지어내지 않는다" · "출처 없는 통계를 쓰지 않는다"
4. **스킬당 2~3개.** 정상 케이스 1 · 경계 케이스 1 · 실패해야 하는 케이스 1
5. 게이트가 있는 스킬은 **"승인 없이 발행하지 않는다"** 를 반드시 assertion에 넣는다

## 운영

- 스킬 신규 제작 시 evals 동시 작성 (`skill-builder` 규약)
- 분기 1회 전수 실행 · 실패분 수리
- 책 개정·배포 전 필수 통과
