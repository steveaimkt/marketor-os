# pm-business-model · 비즈니스 모델 캔버스

marketing-os는 실행·기억·협업은 강하나 사업 전체를 9블록으로 균형 있게 조망하는 힘이 약하다. Business Model Canvas는 가치 창출(좌)·가치 전달(우)·핵심 가치(중앙)·재무 타당성(하)의 흐름으로 사업을 홀리스틱하게 본다. 이미 검증됐거나 운영 중인 사업의 구조 정렬·투자자 커뮤니케이션·전략 점검에 적합하다.

## 언제 호출하나

- "비즈니스 모델 캔버스 그려줘"·"사업 구조 정리하자"·"BMC 만들어줘"·"수익 구조 점검"·"우리 사업 9블록으로"
- 검증된 사업(WMBB 교육·컨설팅·커뮤니티)의 운영 구조를 정렬하거나, 투자자·파트너에게 사업을 설명할 자료가 필요할 때
- 이미 그린 캔버스를 점검·갱신할 때 (gbrain 태그 `business-model` 회수 후 갱신)

경계: 아직 검증 안 된 초기 아이디어의 위험 가정 중심 압축은 pm-lean-canvas가 정본(Problem·Solution·Unfair Advantage 축). 이 스킬은 운영 자원·활동·파트너를 포함한 "돌아가는 사업 전체"를 균형 있게 본다.

## 입력

착수 전 아래를 확인한다. gbrain·프로필에 있으면 다시 묻지 않는다.

1. 대상 사업·제품 라인 (예: WMBB 교육사업 / 컨설팅 / 커뮤니티 / 파트너 TFT)
2. 현재 알고 있는 고객군·가격·주요 비용
3. 기존 자산·파트너 (유튜브 채널·167명 커뮤니티·출강 네트워크·강의 자산)
4. 이 캔버스의 용도 (내부 정렬 / 투자자·파트너 설명 / 전략 점검)

착수 시 먼저 `mcp__gbrain__recall` 또는 `search`로 태그 `business-model`(및 관련 `okr`·`brand`)을 회수해 기존 캔버스·매출 구조·북극성을 읽는다 (read-back).

## 워크플로(캔버스 작성)

같은 입력은 같은 구조를 낸다. 고객에서 출발해 9블록을 채우고 정합성·경제성을 검증한다.

1. **Customer Segments** · 뚜렷한 니즈를 가진 타깃 고객군을 정의한다 (여럿이면 구분).
2. **Value Propositions** · 각 세그먼트의 문제·니즈를 해결하는 핵심 가치. 필요 시 pm-value-proposition 연계.
3. **Customer Relationships** · 관계 방식(개인 응대·셀프서비스·커뮤니티·자동화). 커뮤니티 167명 모델 반영.
4. **Channels** · 발견·구매·전달·지원 경로 (유튜브·LinkedIn·뉴스레터·출강·컨설팅 인바운드).
5. **Key Activities** · 가치 전달의 핵심 프로세스 (콘텐츠 제작·강의 운영·컨설팅 딜리버리·커뮤니티 운영).
6. **Key Resources** · 필요한 물적·지적·인적·재무 자산 (강의 자산·브랜드·채널·팀).
7. **Key Partners** · 사업을 가능케 하는 전략 제휴·공급자 (출강처·파트너 TFT·플랫폼).
8. **Cost Structure** · 고정·변동비와 비용 동인 (제작·인건·광고·플랫폼).
9. **Revenue Streams** · 가격 메커니즘·고객 생애가치 (교육 수강료·컨설팅 fee·커뮤니티 구독·상품화 매출).
10. **정합성 + 경제성 검증** · (a) 9블록이 서로 맞물리는지 정렬 확인 (좌측 자원·활동이 우측 가치 전달을 실제로 지지하는가), (b) 경제성 테스트 LTV > 3배 CAC 여부, (c) 핵심 가정·리스크 문서화.

### ⚠️ 안티패턴 (반드시 피한다)

- **좌우 미정렬** · 우측(가치 전달)이 좌측(자원·활동)으로 뒷받침 안 되면 사업이 안 돈다. 반드시 블록 간 연결을 확인한다.
- **경제성 무검증** · 수익원만 적고 CAC·LTV를 안 보면 캔버스가 아니다. LTV > 3배 CAC를 명시적으로 점검한다.
- **모든 걸 Key로** · 활동·자원·파트너에 다 넣으면 우선순위가 없다. 진짜 "핵심"만 (없으면 뺀다).
- **초기 가설에 오용** · 아직 검증 안 된 아이디어에 이 캔버스를 쓰면 추측이 사실처럼 굳는다 → pm-lean-canvas로.

### 예시 (Steve 비즈니스 · 블록 발췌)

> 대상: WMBB 교육·커뮤니티 사업

- **Customer Segments**: 실행 막힌 1인 마케터·중소 브랜드 마케팅 담당·기업 교육 담당
- **Value Propositions**: "혼자서도 24시간 마케팅팀을 운영하게" · 실행까지 끌고 가는 교육+커뮤니티
- **Channels / Relationships**: 유튜브 유입 → 무료 커뮤니티(167명) → 유료 전환 / 커뮤니티형 관계
- **Revenue Streams**: 교육 수강료 · 컨설팅 fee · 커뮤니티 구독 · 기업 출강
- **경제성**: 유튜브 CAC vs 교육 LTV(재수강·컨설팅 업셀 포함) → LTV/CAC ≥ 3 확인

## 산출물

1. **로컬 파일** · `outputs/{날짜}/pm-business-model/{날짜}-bmc-{사업}.md` — 9블록 표 + 정합성 점검 결과 + LTV/CAC 경제성 판정 + 리스크.
2. **gbrain 적재** · `mcp__gbrain__put_page`로 캔버스 본문 저장 후 `mcp__gbrain__add_tag` 태그 `business-model` 부여. 관련 OKR·북극성 페이지와 `add_link` 연결.
3. **노션** · 요청 시 캔버스를 노션 페이지/DB 행으로 미러링 (프로젝트관리 DB).
4. **검토 요약** · 무엇을·왜·한계(미검증 블록·경제성 취약점)를 3줄로 붙이고, "이 구조로 갈까요? 정렬이 약한 블록은?" 피드백을 청한다.

대외 공유·투자자 제출·파트너 설명 등 외부로 나가는 산출물은 승인 게이트(⏸)를 거친다. 산출 후 취약 블록을 보강하는 후속(pm-okrs로 목표 정렬·voc-analyzer로 세그먼트 검증)을 제안한다.
