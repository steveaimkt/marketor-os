# pm-retro · 회고 (Retrospective)

끝난 작업을 감상으로 흘려보내지 않고 **다음에 실제로 바뀔 행동 2~3개**로 응축하는 스킬. "누구 탓"이 아니라 "무엇을 바꿀까"로 프레임을 고정하고, 각 액션에 담당과 성공기준을 붙여 다음 스프린트에서 검증 가능하게 만든다.

## 언제 호출하나

- "회고하자"·"레트로"·"스프린트 회고"·"이번 캠페인 돌아보자"·"뭐가 잘됐고 뭘 고칠까"
- 스프린트(pm-sprint-plan) 구간이 끝났을 때 (짝 스킬)
- 캠페인·런칭·TFT 프로젝트를 마친 뒤 학습을 다음으로 넘길 때
- 반복되는 문제(같은 지연·같은 실수)를 팀 차원에서 못박아 끊고 싶을 때

경계: 성과 점검·목표 재설계는 pm-okrs, 다음 구간 계획은 pm-sprint-plan이다. 이 스킬은 **끝난 것에서 학습을 뽑아 액션으로 바꾸는 반성 단계**다.

## 입력

```yaml
scope:                  # 필수 · 무엇을 회고하나
  target: "7월 3주차 파트너 온보딩 스프린트"
  goal_stated: "신규 파트너 1일차 자력 완주"   # 당초 목표
format: "A"             # A=Start/Stop/Continue · B=4Ls · C=Sailboat (기본 A)
metrics:                # 선택 · 커밋 대비 완료
  committed: 20         # 커밋 용량/건수
  completed: 16
raw_feedback:           # 필수 · 참여자 소감·관찰 원재료 (나열식 OK)
  - "온보딩 자동화는 예상보다 빨리 끝남"
  - "GA4 권한 대기로 대시보드 3일 지연"
  - "출강 겹친 주에 코어 작업 몰림"
prev_actions:           # 선택 · 지난 회고에서 정한 액션 (후속 추적용)
  - "코어 작업은 출강 없는 요일에 배치 → 지켜졌나?"
```

착수 시 먼저 `mcp__gbrain__recall` 또는 `search`로 태그 `retro`를 회수해 **지난 회고의 액션이 이번에 실행됐는지**를 먼저 점검한다 (read-back). 액션이 매번 증발하면 회고가 의식(儀式)으로 전락한다.

## 워크플로 (회고)

같은 입력은 같은 구조를 낸다.

1. **포맷 선택** · 목적에 맞게 하나를 고른다 (기본 A).
   - **A · Start/Stop/Continue** — 행동 변화 중심. 시작할 것·멈출 것·유지할 것. (범용 기본)
   - **B · 4Ls** — 학습 중심. Liked(좋았던)·Learned(배운)·Lacked(부족했던)·Longed for(바랐던).
   - **C · Sailboat** — 은유 중심. 바람(추진력)·닻(발목 잡은 것)·암초(앞으로의 위험)·섬(목표).
2. **지난 액션 후속** · prev_actions가 실행됐는지 먼저 확인하고 결과를 기록한다 (미실행이면 원인).
3. **소감 그룹핑** · raw_feedback을 테마로 묶는다 (예: 자원 배분·외부 의존·협업 방식). 중복은 합치고 빈도를 표시한다.
4. **성과 대조** · committed 대비 completed(용량 소화율)를 계산하고, 목표(goal_stated) 달성 여부를 판정한다. 숫자와 소감을 잇는다 (왜 그 숫자가 나왔나).
5. **액션 도출 2~3개** · 테마 중 임팩트 큰 것에서 **다음 구간에 바로 실행할** 액션을 2~3개만 뽑는다. 각 액션에 **담당 + 측정 가능한 성공기준**을 붙인다. (많이 뽑으면 하나도 안 된다)
6. **gbrain put + 로컬 저장** · `mcp__gbrain__put_page`로 회고를 태그 `retro`로 적재(포맷·테마·액션·담당). 로컬 저장 후 노션 미러링은 승인 게이트.

### ⚠️ 안티패턴 (반드시 피한다)

- **비난 회고** · "누가 못했다"는 방어를 부르고 학습을 막는다. 사람이 아니라 시스템·프로세스를 문제로 둔다.
- **액션 없는 회고** · 소감만 나누고 끝나면 다음에 똑같이 반복된다. 담당·기준 붙은 액션 2~3개가 산출의 핵심.
- **액션 과다** · 10개를 정하면 0개가 실행된다. 임팩트 순 2~3개로 제한.
- **지난 액션 미추적** · 매번 새 액션만 만들고 이전 것을 안 보면 개선이 누적되지 않는다.

## 산출물

로컬: `outputs/{YYYY-MM-DD}/pm-retro/{YYYY-MM-DD}-retro.md`

형식 (포맷 A 예):

```markdown
# {대상} 회고 · {날짜}
포맷: A (Start/Stop/Continue)
목표 달성: 부분 달성 (1일차 완주 흐름은 붙었으나 대시보드 미완)
커밋 대비: 16/20pt 소화 (80%)

## 지난 액션 후속
- [지킴] 코어 작업 출강 없는 요일 배치 → 실제로 월·수·금 집중, 효과 있었음

## Start (시작할 것)
- 외부 권한(GA4 등)은 스프린트 시작 전날 미리 요청 — 담당: steve — 기준: 다음 스프린트 착수 시 권한 100% 확보

## Stop (멈출 것)
- 준비 안 된 스토리를 용량 채우려 담기 — 담당: 팀 — 기준: 다음 스프린트 커밋 스토리 전부 ready:true

## Continue (유지할 것)
- 출강 없는 요일에 코어 작업 배치 — 담당: steve

## 핵심 테마
1. 외부 의존이 반복 지연 원인 (빈도 2)
2. 자원 몰림은 요일 배치로 완화됨 (개선 확인)
```

검토 요약(무엇을·왜·한계)을 3줄로 붙이고 "이 액션으로 다음 스프린트 갈까요?" 피드백을 청한다. 도출한 액션은 다음 pm-sprint-plan의 입력(개선 항목)으로 잇는다.
