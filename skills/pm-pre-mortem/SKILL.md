# pm-pre-mortem · 사전부검 (리스크 예방)

실행에 들어가기 전에 "이 프로젝트는 이미 실패했다"고 미리 가정하고, 그 실패의 원인을 거꾸로 역산해 지금 손쓸 수 있는 예방책으로 바꾸는 스킬. 낙관 편향으로 놓치는 리스크를 착수 전에 드러낸다.

## 언제 호출하나

- 신규 론칭·캠페인·수주 착수 직전 ("론칭 전에 뭐가 터질지 점검해줘")
- 큰 의사결정·예산 투입 전 사전 리스크 스캔이 필요할 때
- 계획서·플랜을 만든 뒤, 발행·실행 전 마지막 검증으로
- 트리거 예: "프리모템 돌려줘", "사전부검", "이 계획 뭐가 문제일까", "실패 시나리오 뽑아줘", "리스크 점검"

경계: 이미 끝난 일의 원인 분석은 사후부검(post-mortem)이지 이 스킬이 아니다. 이 스킬은 **착수 전** 전용이다. 계획 전반의 논리·가정 압박이 목적이면 plan-reviewer / perspective-reviewer로 넘긴다 (아래 연계 참조).

## 입력

착수 전 아래를 확보한다 (없으면 사용자에게 1~2개만 질문):

1. **대상**: 무엇을 실행하나 (예: 커뮤니티 무료→유료 전환 론칭, 신규 유튜브 영상 시리즈, 기업 컨설팅 수주 제안)
2. **성공 기준**: 무엇이 되면 성공인가 (예: 전환 30명·구독 유지 3개월, 영상 3편 평균 조회 1만, 수주 계약 체결)
3. **기한·자원**: 언제까지, 예산·인력은
4. **gbrain read-back**: 착수 전 `mcp__gbrain__recall` / `search`로 태그 `premortem`·`risk`의 과거 부검 기록을 조회한다. 같은 유형 론칭에서 이미 드러난 실패 원인이 있으면 그대로 후보에 얹는다 (같은 리스크를 두 번 발견하지 않는다).

## 워크플로(사전부검)

### 1단계 · 실패 가정
성공 기준을 뒤집어 선언한다. "6개월 뒤, 이 프로젝트는 실패했다." 부드럽게 "리스크가 있을 수도 있다"가 아니라, **이미 실패한 것으로 확정하고** 시작한다. 이 확정이 낙관 편향을 깨는 장치다.

### 2단계 · 원인 역산 (브레인스토밍)
"왜 실패했는가"를 거꾸로 캐낸다. 최소 8~12개를 뽑되, 빠지기 쉬운 4개 축을 강제로 훑는다:
- **수요·메시지**: 아무도 안 원했다 / 가치 전달 실패 / 타이밍 어긋남
- **실행·품질**: 산출물 미완·품질 미달 / 일정 붕괴 / 리소스 고갈
- **채널·유통**: 도달 실패 / 알고리즘·플랫폼 리스크 / 광고비 효율 붕괴
- **외부·규제**: 경쟁사 선수 / 규제·정책 / 파트너·의존성 미충족

### 3단계 · 3분류 (원전 계승)
각 원인을 세 버킷으로 분류한다 (phuryn 원전의 Tiger 프레임):
- **호랑이 (Tiger · 실제 리스크)**: 근거·논리로 뒷받침되는 진짜 위협. 즉시 대응 필요.
- **종이호랑이 (Paper Tiger · 과장된 우려)**: 그럴듯하지만 실제 확률 낮음. 자원 투입 대신 "기록만" 해 이해관계자 안심.
- **코끼리 (Elephant · 말 안 하는 걱정)**: 팀이 알면서 안 꺼내는 불편한 리스크. 실행 전 반드시 조사·확인.

### 4단계 · 확률·영향 평가
호랑이·코끼리 각 원인에 확률(상/중/하)과 영향(상/중/하)을 매긴다. 우선순위 = 확률 × 영향.
착수 차단성으로 시급도를 나눈다:
- **착수 차단(Launch-Blocking)**: 실행 전 반드시 해소 (예: 핵심 기능 미완, 규제 위반, 핵심 고객 의존성 미충족)
- **빠른 후속(Fast-Follow)**: 착수 후 30일 내 해소
- **관찰(Track)**: 착수 후 모니터링만

### 5단계 · 예방·완화 액션
상위 리스크(착수 차단 + 확률·영향 높은 호랑이)마다:
- 리스크 한 줄 서술
- 구체 예방/완화 액션 1개
- 책임 주체(누가)
- 완료·결정 기한

### 6단계 · 기록·연계
- `mcp__gbrain__put_page`로 부검 결과를 태그 `premortem`·`risk` + 대상명으로 적재 (다음 유사 론칭이 read-back).
- **관점검수 연계(선택)**: 착수 차단 리스크가 남으면 결과를 **perspective-reviewer**(대표·재무·고객·법무·브랜드 5관점 압박)나 **plan-reviewer**(창업자·투자자 시선 계획 검증)로 넘겨 교차 검증. 사전부검이 "무엇이 터질까"를 뽑으면, 관점검수가 "그 대응이 충분한가"를 압박한다.

예시 대상별 감:
- **커뮤니티 무료→유료 전환 론칭**: 호랑이=무료 유저가 유료 가치 체감 실패 / 코끼리=가격 저항을 아무도 검증 안 함 / 종이호랑이=경쟁 커뮤니티 동시 론칭

- **기업 컨설팅 수주**: 호랑이=의사결정자 니즈와 제안 미스매치 / 코끼리=납기 대비 내부 공수 과소추정 / 종이호랑이=경쟁 제안사 가격 우위
