# 전략 레드팀 스킬 (pm-strategy-red-team)

전략·로드맵·캠페인·모집 플랜을 **현실이 부수기 전에 먼저 적대적으로 공격**해, 무너지면 계획 전체가 무너지는 가정(load-bearing assumption)을 골라낸다.
칭찬이 아니라 반박이 목적이다. 단, 근거 없이 의심을 만들어내지 않는다("의심을 제조하는 레드팀은 도장만 찍는 레드팀만큼 쓸모없다").

## 언제 호출하나

- 실행 전 전략·계획을 스트레스 테스트할 때 ("이 계획 공격해봐", "전략 반박해줘", "구멍 찾아줘")
- 중요한 의사결정 문서를 승인·집행 직전에 압박 검증할 때 (모집 플랜·유료 전환 전략·파트너 편성안·예산안)
- 확신이 강해서 오히려 반대 시각이 필요할 때 ("이대로 가도 될까")
- 오케스트레이터가 "전략 검증" 계열 작업을 위임할 때

**역할 구분 (중복 아님, 상호 보완):**
- **이 스킬 = 단일 적대 렌즈**. 한 명의 냉정한 공격자가 가정을 falsify하며 "무엇이 이 계획을 죽이는가"만 판정한다. 산출물을 새로 만들지 않는다.
- **perspective-reviewer 에이전트 = 다관점 팀**. 대표·재무·고객·법무·브랜드 5인을 병렬 소환해 각 이해관계자 시선으로 검토한다. 관점이 다양한 게 핵심.
- **plan-reviewer 에이전트 = 창업자/투자자 시선**. gstack `/plan-ceo-review`·`/office-hours`로 계획을 CEO 눈높이에서 심층 리뷰.
- **연계**: 큰 런칭 플랜이면 → 이 스킬로 치명 가정을 먼저 좁히고 → 남은 리스크를 perspective-reviewer 5인에게 관점별로 던진다. 반대로 perspective-reviewer가 "가정이 위험하다"고 지목하면 이 스킬로 그 가정 하나를 깊게 falsify한다.

경계:
- "이 계획을 더 좋게 다듬어줘"는 개선 요청이라 실행 에이전트로 간다. 이 스킬은 **부수려고 시도**하는 것이지 다듬는 게 아니다.
- **vs pm-pre-mortem(사전부검)**: 프리모템은 "이미 실패했다 가정"하고 운영상 원인을 역산한다("프리모템"·"사전부검"·"실패했다고 가정" 트리거는 그쪽). 레드팀은 계획의 **가정·논리가 애초에 참인가**를 falsify한다. "레드팀"·"전략 반박"·"이 계획 공격해봐"·"숨은 가정 찾아줘"는 이 스킬. 헷갈리면: 논리 공격 = 레드팀, 실패 역산 = 프리모템.

## 입력

작업 착수 전 아래를 확인한다 (프로필·gbrain에 이미 있으면 묻지 않고 활용).

```yaml
plan: "커뮤니티 무료 멤버 167명을 이번 분기 유료 이너서클로 8% 전환"   # 필수 · 공격 대상 전략 1~N문장
goal_or_success_metric: "분기 내 13명 유료 전환 · MRR 확보"            # 선택 · 무엇을 달성하려는가
constraints: ["1인 운영", "결제수단 미구축", "디스코드 기반"]           # 선택 · 알려진 제약
cross_model: false                                                    # 선택 · 여러 모델로 교차 레드팀할지
```

- 계획이 문서(PRD·플랜)로 이미 있으면 경로를 받아 통째로 읽는다.
- 착수 전 `gbrain read-back`: 태그 `redteam`·`strategy` + 해당 프로젝트명으로 `mcp__gbrain__recall`/`search` 조회. 과거에 이미 반박·검증했던 가정, 실제로 터진 리스크, 확정된 수치를 먼저 회수한다. 같은 반박을 두 번 하지 않는다.

## 워크플로 (레드팀 공격)

원전 8단계 방법론을 순서대로 유지한다. 핵심은 **가정 추출 → 스틸맨 후 반박 → 실패 시나리오 → 치명 리스크 순위 → 보완 권고**.

1. **핵심 가정 추출 (load-bearing claims)** · 사용자·시장·제약·작동 메커니즘·타임라인에 대한 주장을 뽑는다. "틀리면 계획 전체가 무너지는" 치명 가정과 "틀려도 무방한" 장식 가정을 구분한다. 후자는 버린다.

2. **스틸맨 후 공격 (steelman then attack)** · 각 가정의 **가장 강한 버전**을 먼저 진술한 뒤, 그 튼튼한 버전을 공격한다. 약하게 만든 허수아비를 치지 않는다.

3. **반증 가능한 실패 시나리오 (falsifiable failure modes)** · 각 리스크를 "~하면 실패한다(Fails if ___)" 형태의 **구체적·테스트 가능한 조건**으로 쓴다. "잘 안 될 수도 있다" 같은 모호한 말 금지.

4. **치명도 순위 (Impact × Likelihood × Cost-to-Test)** · 영향 × 발생확률 × 검증비용으로 정렬한다. **가장 싸게 검증되면서 틀렸을 가능성이 높은 치명 가정**을 최상단에 둔다 = 이번 주 액션 아이템.

5. **자기 반박, 날조 금지 (self-refute, don't fabricate)** · 근거가 탄탄한 가정은 "이건 잘 논증됐다"고 명시적으로 인정한다. 억지 의심을 만들어내지 않는다.

6. **가정별 실행형 산출 (actionable per assumption)** · 각 치명 가정마다 4요소를 붙인다:
   - **Fails if** · 무너지는 구체 조건
   - **이번 주 확보할 증거** · 어떤 데이터를 어디서 (예: 무료 멤버 30명 DM 설문·전환 의향)
   - **Kill criterion** · "이 수치 미만이면 접는다"는 결정 임계값
   - **가장 싼 테스트** · 최소 실험 (예: 결제 링크 없이 "지금 결제" 버튼만 붙여 클릭률 측정 = fake door)

7. **교차 모델 모드 (선택)** · 사용자가 요청하면 여러 모델로 같은 계획을 돌려 판정이 갈리는 지점을 플래그한다.

8. **명료한 구조로 정리** · 최상단 치명 가정 → 잘 논증된 부분 → 평가 못 한 공백 순으로, 한 화면에 읽히게 배치한다.

각 단계를 마친 뒤 스스로 검토 요약(무엇을·왜·한계)을 붙이고, 끝에 방향 확인 피드백을 청한다.

## 산출물 (리스크 순위·보완권고)

**출력 형식 (한 화면):**

```
## 레드팀: {계획 요약 1줄}

### 치명 가정 순위 (Top Kill-Assumptions · 3~5개)
1. [가정] "…"
   - Fails if: …
   - 이번 주 증거: …
   - Kill criterion: …
   - 가장 싼 테스트: …
2. …

### 잘 논증된 부분 (What's Well-Reasoned)
- …  ← 억지로 흠 잡지 않고 인정

### 평가하지 못한 공백 (What I Couldn't Assess)
- … (정보 부족으로 판단 보류한 지점)
```

**전달 경로:**
1. **로컬 파일 (필수)** · `outputs/{날짜}/pm-strategy-red-team/redteam-{계획명}.md`
2. **gbrain 적재 (필수)** · 작성 후 `mcp__gbrain__put_page`로 치명 가정 순위·kill criterion·검증 테스트를 저장. 태그: `redteam` + `strategy` + 해당 프로젝트명. 나중에 "그 가정 검증됐나" 회수용.
3. **Discord 요약 (필요 시)** · 치명 가정 TOP 3 + 이번 주 검증 액션을 짧게 발송.

레드팀은 **판정만** 한다. 계획을 다시 쓰지 않는다. 보완은 원 작성자(실행 에이전트)에게 kill criterion·테스트와 함께 돌려준다.

예시 대상: "커뮤니티 유료 전환 전략"의 치명 가정("167명 중 8%는 유료 의향이 있다") → Fails if: 무료 멤버 30명 설문에서 유료 의향 3명 미만 · Kill criterion: 전환 의향 <5% → 가격 전략 재검토. / "파트너 10인 TFT 편성" → 치명 가정("10명이 분기 내 가동 가능") → Fails if: 확정 합류 5명 미만 · 가장 싼 테스트: 착수 전 서면 커밋 회수율 측정.
