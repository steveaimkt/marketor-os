# pm-lean-canvas · 린 캔버스

marketing-os는 실행·기억·협업은 강하나 새 아이디어를 "검증 가능한 가설"로 압축하는 힘이 약하다. Lean Canvas는 완성도보다 속도를 우선하는 1페이지 가설 도구로, 신규 제품·수익모델을 빠르게 세우고 가장 위험한 가정부터 실험으로 검증하도록 돕는다. Business Model Canvas가 Partners·Activities·Resources를 두는 자리에 Problem·Solution·Unfair Advantage를 넣은 것이 핵심 차이다.

## 언제 호출하나

- "린 캔버스 그려줘"·"사업 가설 한 장으로 정리해줘"·"이 아이디어 검증해보자"·"lean canvas 만들자"·"신규 수익모델 캔버스"
- 새 제품·서비스·수익 라인(교육 신상품·커뮤니티 유료 티어·컨설팅 패키지·파트너 TFT 사업)의 초기 가설을 세울 때
- 이미 세운 캔버스를 점검·갱신할 때 (gbrain 태그 `lean-canvas` 회수 후 갱신)

경계: 이미 검증된 사업의 운영·구조 전체를 잡는 건 pm-business-model(9블록 Business Model Canvas)이 정본이다. 이 스킬은 "아직 검증 안 된 초기 가설을 빠르게, 위험한 가정 중심으로" 압축한다. 단일 세그먼트 가치제안 문장만 필요하면 pm-value-proposition으로.

## 입력

착수 전 아래를 확인한다. gbrain·프로필에 있으면 다시 묻지 않는다.

1. 대상 아이디어·제품 (예: WMBB 신규 교육상품 / 커뮤니티 유료 이너서클 / 컨설팅 패키지)
2. 겨냥하는 고객·초기 채택자 (얼리어답터가 누구인가)
3. 아는 제약 (예산·팀·시간·기존 자산)
4. 이미 검증된 가정 vs 아직 추측인 가정

착수 시 먼저 `mcp__gbrain__recall` 또는 `search`로 태그 `lean-canvas`(및 관련 `brand`·`gtm`)를 회수해 기존 캔버스·시장조사·VoC를 읽는다 (read-back). 같은 가설을 두 번 세우지 않기 위해.

## 워크플로(캔버스 작성)

같은 입력은 같은 구조를 낸다. 9블록을 아래 순서로 채운다.

1. **Problem** · 고객의 상위 3개 페인과 현재 쓰는 부족한 대안을 적는다. (없으면 pain이 약한 것 → 재고)
2. **Customer Segments** · 타깃 페르소나·얼리어답터·대략의 시장 규모. 1~2개 세그먼트로 좁힌다.
3. **Unique Value Proposition** · 왜 이걸 선택하는지 한 문장 차별점. 필요 시 pm-value-proposition 연계.
4. **Solution** · 각 문제에 대응하는 상위 3개 기능·접근. 문제당 1해결이 원칙.
5. **Channels** · 고객 확보·유통·고객 발굴 경로 (유튜브·LinkedIn·뉴스레터·커뮤니티 167명 등 기존 자산 우선).
6. **Revenue Streams** · 가격 모델·고객 생애가치·성장 가정.
7. **Cost Structure** · 고정·변동비·CAC·핵심 비용 동인.
8. **Key Metrics** · 활성화·리텐션·매출·North Star 지표 중 3~5개.
9. **Unfair Advantage** · 쉽게 복제 못 하는 방어벽 (네트워크 효과·브랜드·IP·전환비용·167명 커뮤니티·출강 실적). "가진 척"이 아니라 진짜 방어벽만.
10. **위험 가정 표면화 + 검증 실험 제안** · 9블록에서 가장 불확실하고 치명적인 가정 2~3개를 뽑아, 각각 저비용 검증 실험(랜딩 테스트·사전 판매·인터뷰 10건)을 제안한다.

### ⚠️ 안티패턴 (반드시 피한다)

- **Problem 없이 Solution부터** · 기능 나열은 캔버스가 아니다. 페인이 검증돼야 해결이 의미 있다.
- **가짜 Unfair Advantage** · "좋은 팀"·"빠른 실행"은 복제 가능. 진짜 방어벽만 적는다 (없으면 "없음"이라 쓰고 만드는 걸 과제로).
- **세그먼트 안 좁힘** · "모든 마케터"는 세그먼트가 아니다. 얼리어답터 1~2군으로 좁힌다.
- **위험 가정 미표면화** · 캔버스만 그리고 실험을 안 제안하면 린 캔버스가 아니다. 가장 위험한 가정 → 실험 연결이 이 스킬의 목적이다.

### 예시 (Steve 비즈니스 · 1블록 발췌)

> 아이디어: WMBB 유료 이너서클(커뮤니티 상위 티어)

- **Problem**: 무료 커뮤니티 167명이 실행까지 못 감·개별 피드백 부재·동료 부재
- **Customer Segments**: 무료 커뮤니티 중 실행 의지 높은 상위 10~15%(얼리어답터)
- **UVP**: "혼자 배우다 멈추지 않게, 실행까지 끌고 가는 마케터 이너서클"
- **Unfair Advantage**: 167명 검증 커뮤니티 + 유튜브 유입 파이프라인 + 출강 실적 기반 신뢰
- **가장 위험한 가정**: "무료 회원의 8%가 유료 전환한다" → 실험: 사전 판매 랜딩 2주, 목표 전환 확인

## 산출물

1. **로컬 파일** · `outputs/{날짜}/pm-lean-canvas/{날짜}-lean-canvas-{아이디어}.md` — 9블록 표 + 위험 가정 2~3개와 각 검증 실험 + 다음 액션.
2. **gbrain 적재** · `mcp__gbrain__put_page`로 캔버스 본문 저장 후 `mcp__gbrain__add_tag` 태그 `lean-canvas` 부여(회수·비교용). 관련 시장조사·VoC 페이지와 `add_link` 연결.
3. **노션** · 요청 시 캔버스를 노션 페이지/DB 행으로 미러링 (프로젝트관리 DB).
4. **검토 요약** · 무엇을·왜·한계(가장 불확실한 블록·검증 안 된 가정)를 3줄로 붙이고, "이 가설로 갈까요? 먼저 검증할 가정은?" 피드백을 청한다.

대외 공유·투자자 제출·팀 공유 등 외부로 나가는 산출물은 승인 게이트(⏸)를 거친다. 산출 후 가장 위험한 가정을 검증하는 후속(랜딩 테스트·인터뷰·trend-scanner·voc-analyzer)을 제안한다.
