# PRD 작성 스킬 (pm-create-prd)

제품/기능 아이디어를 팀·파트너·클라이언트가 합의할 수 있는 **8섹션 PRD 문서**로 만든다.
"뭘 왜 누구를 위해 언제까지"를 검증 가능한 형태로 고정하는 것이 목적이다.

## 언제 호출하나

- 새 기능·상품을 만들기 전 스펙을 문서로 확정할 때 ("PRD 써줘", "요구사항 정리")
- 컨설팅·출강 납품물의 범위와 성공 기준을 클라이언트와 합의해야 할 때
- 파트너 TFT(비주얼인프라·피알원 등) 협업에서 서로의 기대치를 한 장으로 맞출 때
- 커뮤니티(167명)·유튜브 채널의 신규 기능을 기획 단계에서 문서화할 때
- 오케스트레이터가 "제품기획" 계열 작업을 위임할 때 (product-lead)

경계: 이미 만들 것이 정해졌고 "만들기"만 남았으면 PRD가 아니라 실행 에이전트로 간다. 이 스킬은 **결정 이전, 합의를 위한 문서**를 만든다.

## 입력

작업 착수 전 아래 3가지를 먼저 확인한다 (프로필·gbrain에 이미 있으면 묻지 않고 활용).

```yaml
product_name: "이너서클 유료 전환 기능"     # 필수 · 무엇에 대한 PRD인가
problem: "무료 커뮤니티 167명이 유료로 넘어갈 계기가 없다"  # 필수 · 해결할 문제 1문장
audience_or_stakeholders:                   # 필수 · 누구를 위해 / 누가 합의자인가
  - "유튜브 유입 무료 멤버 (전환 후보)"
  - "Steve (의사결정)"
timeframe_hint: "이번 분기"                   # 선택 · 상대적 기간 (정확한 날짜 지양)
constraints: ["1인 운영", "디스코드 기반", "결제 수단 미구축"]  # 선택
```

- 입력이 얇으면 채워 넣지 말고 **가정으로 표시**(⚠️ 검증 필요)한 뒤 진행한다.
- 착수 전 `gbrain read-back`: 관련 슬러그·태그(`prd`, 해당 프로젝트명, `커뮤니티`/`교육상품` 등)를 `mcp__gbrain__recall`/`search`로 조회해 과거 결정·수치·제약을 먼저 회수한다. 같은 걸 두 번 묻지 않는다.

## 워크플로 (8섹션 PRD)

원전 8섹션 구조와 각 섹션의 핵심 규율을 그대로 유지한다. 순서대로 채운다.

**작성 원칙 (전 섹션 공통)**
- 초등학교 졸업자도 읽을 수준으로 · 전문용어 대신 짧고 분명한 문장
- 구체적·데이터 기반 · 추정은 반드시 "⚠️ 가정 · 검증 필요"로 표시
- 간결하되 빠짐없이

1. **Summary (요약)** · 이 문서가 다루는 것을 2~3문장으로. "무엇을, 누구를 위해, 왜 지금".

2. **Contacts (이해관계자)** · 이름·역할·비고를 표로. 누가 결정하고 누가 실행하고 누가 검수하는지 명시. (컨설팅/파트너 PRD면 클라이언트 측 의사결정자 필수)

3. **Background (배경)** · 맥락과 범위 · "왜 지금인가"(타이밍) · 최근 무엇이 바뀌었거나 가능해졌는가. (예: 유튜브 유입이 167명에 도달해 임계 규모가 생김)

4. **Objective (목표)** · 핵심 목표와 그 중요성 · 회사와 고객 각각의 이득 · 상위 전략(북극성: 매출 5억→100억)과의 정렬 · 성공 지표는 **SMART OKR 형식**(구체적·측정가능한 목표치). 예: "분기 내 무료 멤버의 8%를 유료 이너서클로 전환 (13명)".

5. **Market Segment(s) (시장 세그먼트)** · 누구를 위해 만드는가 · 기존 제약. ⚠️ **인구통계가 아니라 '문제'와 'Jobs-to-be-done'으로 정의**한다. ("30대 남성" X → "혼자 마케팅을 돌리다 막힐 때 물어볼 동료가 없는 사람" O).

6. **Value Proposition(s) (가치 제안)** · 어떤 고객 과업/니즈를 푸는가 · 고객이 얻는 것 · 없애주는 페인 · 경쟁 우위. 필요 시 Value Curve로 차별점 시각화.

7. **Solution (솔루션)**
   - **7.1 UX/프로토타입** · 사용자 흐름·와이어프레임 (텍스트 플로우로도 가능)
   - **7.2 핵심 기능** · 기능별 상세 (무엇을 하는가 · 왜 필요한가)
   - **7.3 기술** · 전략적으로 중요할 때만 (결제·MCP·자동화 연결 등)
   - **7.4 가정** · 검증이 필요한 미검증 믿음 목록 (⚠️ 표시)

8. **Release (릴리스)** · 소요 기간은 **상대적 기간**으로(정확한 날짜 지양) · MVP와 이후 반복을 구분 · 단계별 롤아웃 계획.

각 섹션을 채운 뒤 스스로 검토 요약(무엇을·왜·한계)을 붙이고, 끝에 방향 확인 피드백을 청한다.

## 산출물

1. **로컬 파일 (필수)** · `outputs/{날짜}/pm-create-prd/PRD-{제품명}.md`
   - 8섹션 전체 · frontmatter에 product_name·작성일·상태(draft/review/approved) 기록
2. **gbrain 적재 (필수)** · 작성 후 `mcp__gbrain__put_page`로 PRD 요지(목표 OKR·세그먼트·MVP 범위)를 저장. 태그: `prd` + 해당 프로젝트명. 후속 작업이 회수할 수 있게 한다.
3. **노션 페이지 (필요 시)** · 팀·클라이언트 공유가 필요하면 노션 페이지로 미러링.

⏸ **승인 게이트** · 대외 공유용 PRD(클라이언트 납품·파트너 TFT 공유·유료 상품 스펙)는 **발행/전달 전 Steve 승인**을 받는다. 로컬 draft 작성까지는 자율, 외부로 나가는 순간은 반드시 확인.
