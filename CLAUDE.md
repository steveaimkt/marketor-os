# marketing-os · 프로젝트 헌법

이 파일은 Claude Code가 `marketing-os/` 디렉토리에서 작업할 때 항상 로드됩니다.
모든 에이전트·스킬·명령은 아래 원칙을 따릅니다.

## 프로젝트 정체성

**이름**: marketing-os (마케팅 OS)
**목적**: 1인 마케터가 24시간 가동되는 30개 에이전트 팀을 운영할 수 있도록 하는 Claude Code 기반 자동화 시스템
**강의**: 패스트캠퍼스 「24시간 Full 가동 마케팅 자동화! 30개 에이전트로 완성하는 클로드코드」
**구조**: 메인 오케스트레이터 1개 + Part 3~9 서브에이전트 30개 + 핵심 스킬 5개 + MCP 12개

## 협업 프로토콜 (모든 작업 공통 · 이 OS의 핵심 동작)

이 OS는 일반 마케터가 명령어를 몰라도 쓸 수 있어야 한다. 모든 작업에서:

1. **자연어 우선**: 사용자가 자연어로 요청하면 세팅된 장치(스킬→에이전트→커맨드→MCP)를 트리거 매핑으로 **알아서 발동**한다. 명령어를 외우라고 요구하지 않는다.
2. **시작 전 확인**: 작업 착수 전 결과를 좌우하는 핵심 질문 1~3개를 먼저 묻는다 (이미 답이 프로필·메모리에 있으면 묻지 않고 활용).
3. **방향 제시**: 무작정 실행하지 않고 추천안 + 근거를 짧게 제시한 뒤 진행한다.
4. **결과 검토**: 산출물 완성 시 스스로 검토 요약(무엇을·왜·한계)을 붙인다.
5. **피드백 요청**: 끝에 반드시 피드백을 청한다 ("이 방향 맞나요? 고칠 점은?").
6. **기억**: 받은 피드백·확정된 선호·반복 패턴을 `brand/marketer.md` 성장 로그와 메모리에 적재한다. **같은 걸 두 번 묻지 않는 것**이 성장의 증거다.
7. **장치 제안**: 반복되는 업무를 감지하면 스킬화(skill-builder)·자동화(cron/루틴)를 먼저 제안하고, 필요한 스킬/워크플로/MCP가 없으면 찾아서 제안→설계→구축한다 (구축은 사용자 승인 후).
8. **운영 면**: 로컬은 Claude Code(터미널·데스크탑), 외부는 Discord(트루먼 봇)에서 동일하게 동작한다.

### 업무 착수 전 팀 판정 (필수 · 2026-07-20 steve 지시)

모든 업무 요청은 착수 **전에** `agents/TEAM-MODE.md §1` 표로 팀 여부를 판정한다. 판정을 건너뛰지 않는다.

| 업무 성격 | 처리 | 예 |
|---|---|---|
| **기획·전략·검수·리서치·다관점 판단** | **팀 (기본값)** · 편성안 1줄 고지 후 **즉시 소집** (승인 대기 안 함) | 콘텐츠 기획·캠페인 설계·플랜 검수·시장 스캔·원인 규명 |
| **실행·수리·조회·단순 변환** | **단독** · 팀 만들지 않는다 | 파일 수정·경로 복구·동기화·검색·포맷 변환 |
| 애매 | **팀** · 단, 편성 이유를 한 줄 밝힌다 | |

- 편성은 `TEAM-MODE.md §2` 표준 3종(부서 협업팀·검수 토론 패널·리서치 스웜)에서 고른다. 새 편성을 즉흥으로 만들지 않는다.
- 실행은 `skills/team-run`. 도메인 오케스트레이터(`youtube-content-orchestrator`·`campaign-orchestrator` 등)가 있으면 **그쪽이 우선**이다.
- 토큰 소모가 큰 편성(5명 이상)은 고지에 규모를 함께 적는다.
- ⚠️ **사고 기록**: 2026-07-20 유튜브 콘텐츠 기획을 `youtube-content-orchestrator` 없이 단독 처리 → 경쟁 채널 실측·트렌드 리서치가 빠진 기획이 산출됨. 이 규칙은 그 재발 방지다.

#### ⚠️ 알려진 결함 · 팀 결과 미반환 (2026-07-20 확인, 미해결)

**증상**: 스폰한 팀원이 작업 후 최종 보고를 반환하지 않고 `idle_notification`만 반복 전송. `SendMessage`로 재요청해도 무응답. 2026-07-20 두 건에서 재현 (데스크탑 스캔 1명 · 100억 검수 패널 4명 전원).

**원인**: 에이전트 정의 문제 아님(`perspective-reviewer` frontmatter 정상). 결과 반환 계층으로 추정. 미규명.

**✅ 우회로 (2026-07-20 실전 검증 완료)**

서브에이전트에게 **"결과를 반환하지 말고 먼저 파일로 써라"** 고 지시하면 작업이 성립한다.

```
산출 경로: outputs/{YYYY-MM-DD}/{도메인}/{에이전트명}-{대상}.md
```

검증: `acquiring-editor`가 통합원고(478KB) 블라인드 판정 후 9,757바이트 보고서를 정상 생성. **반환은 여전히 끊겼지만 파일은 남았다.** 즉 결함은 **반환 단계에만** 있고 작업 수행 자체는 정상이다.

→ **규약 정본은 `agents/_conventions.md §I`(산출물 착지)** · 2026-07-23 기준 **에이전트 71개 전체에 인라인 배선 완료**. 새 에이전트를 만들 때도 이 블록을 반드시 넣는다(`agents/part4-research/voc-analyzer.md` 하단이 표준 형태 · 리더급은 유휴 2회 확인 규칙 1줄 추가).

**그 외 규칙**
1. 팀 소집 후 **결과가 오지 않으면 기다리지 말 것.** 유휴 알림 2회 = 반환 실패로 간주하고 **먼저 산출 파일을 확인**한다. 파일이 있으면 성공이다.
2. 파일도 없으면 단독 수행으로 전환하고 **사용자에게 팀 실패를 먼저 알린다.** 팀이 한 것처럼 보고하지 않는다.
3. **근거 데이터는 소집 전에 리더가 직접 확보**해 둔다. 그래야 폴백이 즉시 가능하다.
4. 이 결함이 해소되면 이 절을 삭제한다.

## 도구 우선순위 (위에서 아래로)

## 명명 규칙

- **에이전트 이름**: kebab-case, 영문, 동사+명사 (예: `competitor-monitor`, `ga4-html-report`)
- **스킬 이름**: kebab-case, 영문 (예: `newsletter-writing`)
- **슬래시 명령**: `/<동사>-<목적어>` (예: `/research-competitor`, `/weekly-report`)
- **산출물 파일**: `{날짜}-{종류}.{ext}` (예: `2026-05-15-meta-roas-report.html`)

## 산출물 표준

모든 에이전트의 최종 산출물은 **다음 3곳 중 하나 이상**에 도달해야 합니다:
1. **Discord 채널** · 짧은 요약 + 링크 (webhook 사용)
2. **Notion 페이지** · 전문 + 인사이트 (장기 보관)
3. **로컬 파일** · `outputs/{날짜}/{에이전트명}/` (감사용)

## 보안 원칙

- `.env`는 절대 커밋하지 않음 (`.gitignore`에 등록됨)
- 광고 집행 권한이 있는 에이전트(`meta-ads-analyzer` 등)는 **읽기 전용 토큰**을 기본값으로 사용
- 광고 예산 변경·콘텐츠 발행은 **사용자 승인** 단계를 반드시 거침 (디스코드 reaction 또는 명시적 명령)
