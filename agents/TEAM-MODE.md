# TEAM-MODE · 에이전트 팀 운영 규약 (실험 기능)

> `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` (.claude/settings.json에 활성화됨 · Claude Code v2.1.178+).
> 서브에이전트(결과만 보고)와 달리, 팀원은 **독립 세션 + 공유 작업 목록 + 상호 메시징**으로 자체 조율한다.
> 트루먼(메인 세션)이 항상 리더. 이 문서는 트루먼이 팀을 소집·해산하는 기준과 규칙의 정본이다.

## 1. 언제 팀인가 (판단 기준)

| 상황 | 모드 | 이유 |
|---|---|---|
| 단건 업무·순차 체인(커맨드 이어달리기) | **서브에이전트** (기본) | 결과만 필요 · 팀은 조율 오버헤드·토큰 낭비 |
| 여러 부서가 **서로 결과를 주고받으며** 동시 작업 (예: 신제품 런칭 준비) | **팀** | 팀원 간 직접 메시징으로 리더 경유 없이 조율 |
| 중요 산출물 **다관점 토론 검수** (런칭 플랜·예산안) | **팀** | 서로의 결론을 반박하는 토론 구조 · 앵커링 방지 |
| 원인 불명 문제의 **경쟁 가설 조사** (성과 급락 원인 등) | **팀** | 가설별 독립 조사 + 상호 반박 → 생존 가설이 정답 |
| 같은 파일을 여러 명이 수정해야 하는 작업 | **서브에이전트 순차** | 팀원 동시 편집 = 덮어쓰기 사고 |

기본값은 서브에이전트. **팀 소집은 사용자 승인 후에만** (트루먼이 "팀으로 돌릴까요? 토큰 소모 큼" 고지 → ⏸).

## 2. 표준 편성 3종

| 편성 | 구성 (기존 정의 재사용) | 용도 · 트리거 예 |
|---|---|---|
| **부서 협업팀** (3~5명) | 관련 부서장: research-lead·content-lead·ads-lead·commerce-lead 중 해당 부서 | "신제품 런칭 준비 팀으로" · 부서별 산출물을 서로 인계하며 동시 진행 |
| **검수 토론 패널** (3~5명) | perspective-reviewer ×N (각자 다른 관점: 대표·재무·고객·법무·브랜드) | "이 플랜 팀 토론 검수" · 서로의 판정을 반박, 합의만 생존 |
| **리서치 스웜** (3명) | trend-scanner·competitor-monitor·voc-analyzer 계열 | "시장 전방위 스캔" · 발견을 서로 공유·교차 검증 |

- 팀원 생성 시 **기존 agents/ 정의를 타입으로 지정** ("Spawn a teammate using the ads-lead agent type…") · tools 허용목록·model이 그대로 적용됨
- ⚠️ 정의의 `skills`·`mcpServers` frontmatter는 팀원에게 **미적용** · 팀원은 프로젝트 설정(.mcp.json)에서 MCP를 로드하므로 실사용엔 문제 없음
- 이름은 예측 가능하게: 부서명 그대로 (`ads`, `content`, `research`…) · 이후 "ads에게 물어봐"로 지목 가능

## 3. 운영 규칙 (marketing-os 헌법과의 접점)

1. **결재는 리더로 버블업** · 팀원의 권한 프롬프트는 트루먼 세션에 뜬다. 발행·예산 도구는 어차피 PreToolUse 훅(require-approval)이 잠그므로 팀원이 우회 불가.
2. **mutating 작업 팀원은 계획 승인 필수** · "Require plan approval before changes" 지시. 승인 기준을 프롬프트에 명시 (예: "발행 예약이 포함된 계획은 거부하고 나에게").
3. **파일 소유권 분리** · 팀원별 산출물은 `outputs/{날짜}/{팀원명}/`에만 쓴다. 공용 파일(brand/·CLAUDE.md) 수정은 리더 전담.
4. **게이트는 팀 밖** · gate-auditor(⛔ 차단 권한)는 팀원으로 넣지 않는다. 팀 완료 후 리더가 별도 서브에이전트로 호출 (차단 권한 단일화).
5. **gbrain 규약 유지** · 각 팀원도 read-back→작업→put(태그). 팀 해산 전 리더가 종합 결과를 build-log 1행 + gbrain 기록.
6. **작업 크기** · 팀원당 작업 5~6개 유지, 팀원 3~5명 시작. 15개 독립 작업이면 3명이 적정.
7. **방치 금지** · 리더는 진행 모니터링·재지정·종합이 본업. 팀 가동 중 리더가 직접 구현에 뛰어들지 않는다.

## 4. 소집 발화 예시 (트루먼이 내부적으로 쓰는 형식)

```
신제품 런칭 팀: research-lead·content-lead·ads-lead 타입으로 팀원 3명을 생성.
이름은 research/content/ads. 각자 담당: 시장 검증 / 메시징·캘린더 / 매체 플랜.
서로의 중간 산출물을 메시지로 공유하며 진행하고, outputs/{날짜}/{이름}/에만 쓸 것.
발행·예산 변경이 포함된 계획은 승인 요청으로 올릴 것.
```

## 5. 제한 사항 (실험 기능 · 사용자에게 고지할 것)

- 세션당 팀 1개 · 중첩 팀 불가 · 리더 고정 (트루먼 승격/이양 불가)
- `/resume` 후 in-process 팀원 복원 안 됨 → 재소집 필요
- 작업 상태 지연 가능 → 막힌 듯하면 리더가 상태 수동 갱신
- 토큰 소모 = 팀원 수에 비례 (일상 업무는 서브에이전트가 정답)
- 표시 모드: 기본 in-process (에이전트 패널 ↑↓+Enter). 분할 창 원하면 `claude --teammate-mode auto` (tmux/iTerm2)

## 6. 하네스 6패턴 매핑 (revfactory/harness 표준 언어 · 2026-07-17 보강)

> 전역 스킬 `~/.claude/skills/harness`(팀-아키텍처 팩토리) 설치됨. marketing-os는 6패턴을 이미 전부 사용 중 · 새 업무 설계 시 아래 표의 언어로 소통한다.

| 하네스 패턴 | marketing-os 구현 | 새 업무에 쓸 때 |
|---|---|---|
| Supervisor | 트루먼 라우팅·결재 접수 | 항상 (리더 고정) |
| Hierarchical Delegation | 트루먼 → 부서장 → 실무자 | 카테고리 업무 위임 |
| Pipeline | 커맨드 체인 (~돌려줘) + 핸드오프 계약 | 순서 고정·산출물 릴레이 |
| Fan-out/Fan-in | 부서 협업팀 · 병렬 부서장 소환 | 독립 조사 후 종합 |
| Producer-Reviewer | 실무자 → gate-auditor·quality-reviewer-6axis | 발행물 전부 (gate:true) |
| Expert Pool | 검수 토론 패널 (perspective-reviewer ×5 관점) | 중요 의사결정 압박 검토 |

- **팀 재구성 패턴** (하네스 준용): Phase마다 다른 전문가 조합이 필요하면 이전 팀 산출물을 파일로 저장 → 팀 해산 → 새 팀 구성. 산출물은 `outputs/{날짜}/`에 보존되므로 새 팀이 Read로 인계받는다 (세션당 활성 팀 1개 제약의 우회로)
- **QA 커버리지 맵**: 하네스 QA 가이드의 "경계면 불일치" 검증은 기존 도구가 분담 · 경로 경계면=check-links · 정본↔배치판=check-org-drift · 스킬 계약=msk validate.py · 레지스트리=validate-agents. 미커버 갭(산출물 런타임 검증: HTML 리포트 실제 렌더 등)은 대기열 후보로 기록

## 7. 검증 방법 (다음 세션에서)

1. 새 세션 시작 (env 반영) → "팀 기능 켜졌어?" 확인
2. 소규모 리허설: "리서치 스웜 3명으로 '클로드 교육' 시장 훑어줘" → 에이전트 패널에 3명 표시 확인
3. 팀원 간 메시징·작업 목록(Ctrl+T) 동작 확인 → 이상 없으면 부서 협업팀 실전 투입
