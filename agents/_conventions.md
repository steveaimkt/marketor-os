# 에이전트 공통 규약 (Conventions)

> 이 파일은 개별 에이전트에 반복 서술하지 않기 위한 **단일 참조원**이다.
> 에이전트 하단의 "공통 규칙" 블록은 이 파일 A~H를 가리킨다.

## A. 브레인·효율 (gbrain)
- **read-back(작업 전)**: `mcp__gbrain__*`로 과거 맥락 조회. **브랜드명/주제 태그로 회수** · 태그가 회수를 만든다. 백지 시작 금지.
- **put(작업 후)**: 산출물·인사이트를 **같은 태그**로 기록 → 다음 작업이 누적 위에서 시작(복리).
- **롤백**: 산출물이 게이트에서 반려되면 **마지막 정상본(last-known-good)** 으로 되돌려 재시도(수동 버전관리 금지).
- 없거나 무응답이면 무시하고 진행(best-effort).
- **병렬**: 서로 의존 없는 조회·수집은 동시 실행(팬아웃).
- **모델 티어링**: 수집·조회 = 경량 / 판단·합성·창작 = opus.
- **컨텍스트 절약**: 서브 결과는 structured 요약만 반환(원문 덤프 금지).

## B. 핸드오프 계약 (Handoff Contract) · "no agent starts cold"
에이전트가 다른 에이전트로 작업을 넘길 때, 프롬프트 한 줄이 아니라 **계약**을 넘긴다.
헤더의 `chains_to`는 **방향**, 아래 4블록은 **화물**이다.

```
HANDOFF → <받는 에이전트>
- Context     : 완료한 것(구체적으로) / 산출물 경로·gbrain 태그 / 브랜드=profile 참조 / 제약
- Deliverable : 다음이 만들 것 + 측정가능 인수기준(체크박스 □)
- Quality     : 완료 증명(수치·근거 파일). "좋아 보임" 같은 주관 판정 금지.
- Gate        : 대외 발행물이면 어느 게이트를 통과해야 하는지 명시
```

받는 에이전트는 Context 4항이 채워져 있으면 **재질문 없이** 시작한다.
이것이 "스킬 모음이 아니라 OS"를 증명하는 배선이다.

## C. 가동 모드 (Activation Modes) · 100개 중 몇 개를 켜나
트루먼이 intake에서 스코프를 보고 모드를 먼저 정한 뒤 로스터를 확정한다.

| 모드 | 켜는 범위 | 예 | 상업 층 매핑 |
|---|---|---|---|
| **Micro** | 1개 에이전트 즉답 | "리뷰 분석" → voc-analyzer | 오픈코어/단건 |
| **Sprint** | 3~6개(1 Part 또는 체인) | "이번 주 카피" → ad-copy-ab + quality-reviewer-6axis + content-calendar | 플러그인 팩 |
| **Full** | 전체 팬아웃 4페이즈 | `/ax-team-run weekly` | Full 구독 |

## C-2. 정본 스킬 (canonical_skill)
헤더에 `canonical_skill: "0XX"`가 있는 에이전트는 **marketing-100-skills의 해당 SKILL.md가 방법론 정본**이다.
작업 시작 시 정본을 Read해 Contract·Phases를 적용한다. ⛔복사 금지 · 참조만. 경로·절차는 orchestrator §3.7.

## D. 게이트 기본값 (Gate Model) · defaults to NEEDS WORK
- 품질 게이트(`quality-reviewer-6axis`)는 **기본값이 "통과"가 아니라 "반려(NEEDS WORK)"**.
  수치·근거가 제시되어야 통과. 주관 판정("괜찮다")으로는 못 넘어간다.
- 대외 발행물(헤더 `gate: true`)은 **compliance 게이트**(약사법·표시광고법·화장품법) 통과 의무.
- 검수 루프는 **최대 3회 재시도**, 초과 시 사람에게 에스컬레이션.

## E. 업무 브리핑 게이트 (Briefing Gate) · 만들기 전에 합의한다
> marketing-100-skills TEAM.md "업무 브리핑 게이트" 이식(2026-07-16). 결과물을 만들고 보고하는 게 아니라, **무엇을 어떻게 만들지 먼저 1화면으로 합의**한다.

의미 있는 실행 전, 담당 에이전트 명의로 **브리핑 1화면**을 제시하고 ⏸승인 후 착수:
```
① 진행 계획: {단계 요약 · 게이트 포함 여부}
② 예상 산출물: {스킬/에이전트 Output Format·outputs 필드에서 도출한 형태}
③ 필요한 입력: {정보 역요청 · ⑴구하는 방법(앱 경로)까지 ⑵없으면 쉬운 문답 3~5 ⑶그것도 없으면 [추정] 사전고지+실측 재검 예약}
⏸ 이대로 진행할까요?
```
- **한 화면 = 한 결정** · ⏸ 답 전 실행 금지. 체인이면 단계별 표 + ⏸ 계획 전체 1회 + mutating 단계마다 추가.
- 사용자가 **"바로 해줘"** 명시 시 브리핑 생략 가능 · 단 mutating(발송·발행·집행) ⏸는 생략 불가.
- 입력이 충분하고 요청이 구체적이면 3줄로 축약 가능(생략 불가).
- **사문화 방지**: 사용자가 "바로 해줘"를 3회 연속 쓰면 → "기본을 축약 브리핑으로 바꿀까요?" 1회 제안(규칙이 죽은 채 방치되지 않게).
- **직함은 한국어로** · 영문 에이전트명·스킬 ID·파일명은 사용자 화면에 노출 금지(내부 로그에서만).

## F. 결과 확인 + 개선 루프 (전달 직후 의무)
1. **결과 확인 게이트**: 산출물 전달 직후 ⏸ "보완할 점 없나요?" · 지적은 같은 담당이 재브리핑 없이 즉시 반영. 브리핑 예상과 다르게 나온 부분은 먼저 자백.
2. **지적이 규칙으로**: 같은 유형 지적 2회 누적 → 해당 스킬 Anti-Patterns 갱신 제안(정본은 100-skills, 제안만).
3. **개선 질문**: 통과 후 "다음에 이런 업무 때 뭘 개선하면 좋을까요?" 1회(선택 답변) → gbrain 개선 대기열 적재. 무응답이면 조용히 진행(재촉 금지).

## G. 실데이터 우선 원칙 (2026-07-15 사장 지시)
- 세션에 실측 도구가 연결돼 있으면(GA4 Python 경로·네이버 데이터랩·YouTube·검색 MCP 등) **추정 모드 금지 · 반드시 실측 먼저**.
- 추정은 도구 부재·조회 실패 시의 **최후 폴백**이며, 그 경우 산출 **전에** "실측 불가 · 추정으로 진행할까요?" ⏸ 고지 의무.
- 추정치를 실측처럼 보이게 하는 것은 **게이트 위반과 동급**. 모든 추정치는 `[추정]` 태그.

## H. 권한 분리 & 협업 (Authority Separation)
- **제작(부서장·실행 에이전트) / 품질 판정(`quality-reviewer-6axis` 6축) / 규제 판정(`staff/gate-auditor` 유일 ⛔) / 관점 검수(`staff/perspective-reviewer` 5인 병렬, 심층은 `part10/plan-reviewer` gstack)는 상호 대행 불가.** ⛔ 번복은 사용자만.
- **단일 보고선**: 서브에이전트는 트루먼에게만 structured 요약으로 보고. 사용자 대면 문장은 트루먼이 조립.
- **피드포워드**: 앞 에이전트 산출물 파일이 다음의 입력(§B 핸드오프 계약). 사용자에게 재설명 요구 금지.
- **모델 티어링**: 판정 전용·병렬 소환 에이전트(plan-reviewer 등)는 `model: sonnet` 권장(산출물 미생성이라 품질 리스크 최소, 비용 최대 절감).
- **규제 3중 접점 호출 순서(고정)**: ①제작 중 예비 = 6축의 규제안전성 축(자문) → ②중요 산출물 사전 = 법무 관점 검수(선택) → ③발행 직전 최종 = **gate-auditor ⛔(구속력, 유일)**. 셋은 순서지 대체재가 아니다.

## I. 산출물 착지 (Landing) · 반환보다 파일이 먼저다
> §H "단일 보고선"의 **폴백**. 2026-07-20 확인된 팀 결과 미반환 결함의 검증된 우회로다.
> 결함은 **반환 단계에만** 있고 작업 수행 자체는 정상이므로, 파일로 착지시키면 업무가 성립한다.

**표준 산출 경로**
```
outputs/{YYYY-MM-DD}/{도메인}/{에이전트명}-{대상}.md
```

**팀원(서브에이전트) 측 규칙**
1. 팀원으로 스폰되면 **결과를 반환하려 하지 말고 먼저 위 경로에 파일로 쓴다.**
2. 파일 쓰기가 **완료된 뒤에만** 요약을 반환한다. 반환이 실패해도 파일은 남는다.
3. 파일에는 판단 근거·수치·한계를 모두 담는다. "요약은 반환으로 보내겠다"는 전제 금지.

**리더(트루먼·오케스트레이터) 측 규칙**
4. 소집 후 결과가 오지 않아도 **기다리지 않는다.** 유휴 알림 2회 = 반환 실패로 간주하고 **먼저 산출 파일을 확인**한다. 파일이 있으면 성공이다.
5. 파일도 없으면 단독 수행으로 전환하고 **사용자에게 팀 실패를 먼저 알린다.** 팀이 한 것처럼 보고하지 않는다.
6. **근거 데이터는 소집 전에 리더가 직접 확보**해 둔다. 그래야 폴백이 즉시 가능하다.

> 검증: `acquiring-editor`가 통합원고(478KB) 블라인드 판정 후 9,757바이트 보고서 정상 생성. 반환은 끊겼지만 파일은 남았다.
> ⚠️ 이 결함이 해소되면 1~2·4~5를 삭제하고 §H 단일 보고선으로 복귀한다. 경로 규약(위 3·6)은 그때도 유지.
