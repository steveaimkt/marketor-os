# 100 멀티 에이전트 오케스트레이션

이 OS의 종착점. 스킬 99개를 도구로 쓰는 단계를 지나, **하나의 마케팅 OS로 조직화하는 방법**을 설명하고 설계해준다. 이 스킬이 다루는 실물이 곧 이 OS 자신이다 — 루트 SKILL.md(오케스트레이터)와 TEAM.md(부서장 10+스태프 3)의 구조를 해설하고, 사용자 업무에 맞는 팀 구성을 설계한다.

## Contract
1. 아키텍처 설명은 실제 파일 구조(루트 SKILL.md · ROUTING.md · TEAM.md · gates/ · logs/)를 근거로 한다 — 존재하지 않는 가상 구조를 지어내지 않는다.
2. 라우팅 원리는 Progressive Disclosure로 설명된다: 상시 컨텍스트는 ROUTING.md 한 장, 개별 SKILL.md는 매칭 시에만 로드.
3. 팀 구성안의 모든 요소는 실존 식별자로 매핑된다 — 부서장은 TEAM.md의 10명 중에서, 스킬은 ROUTING.md ID로, 커맨드는 대표 커맨드 15종에서.
4. `mutating: true`(발송·발행·집행) 스킬과 `gate: true` 발행물의 승인 정책(⏸/gate-auditor 경유)이 팀 구성안에 반드시 포함된다 — 자동화 설계에서 사람 게이트를 없애지 않는다.
5. 상시 가동(스케줄) 제안은 수동 운영으로 산출물이 안정된 스킬에 한한다 — 첫 설계부터 전 자동화를 권하지 않는다.

## Phases
1. **아키텍처 투어 — 요청 1건의 생애주기** — 루트 SKILL.md의 7단계를 사용자 예시 1건으로 시연 설명: INTENT(ROUTING 매칭) → CONTEXT(profile.md 주입) → PLAN(체인 승인 ⏸) → EXECUTE(팀 위임) → GATE(compliance) → LOG(build-log 1행) → HANDOFF(chains_to 추천). "왜 100개를 다 안 올리는가"(Progressive Disclosure)와 "왜 브랜드를 재질문 안 하는가"(이중 허브: profile.md+my-playbook.md)를 함께 짚는다.
2. **조직도 해설 — TEAM.md** — 사용자(CEO·최종 결재) → 오케스트레이터(팀장·라우팅과 조립) → 부서장 10(카테고리별 001~100 분담) + 스태프 3(perspective-reviewer 5관점 병렬 검수 · gate-auditor 유일한 ⛔ 권한 · skill-builder 제작 전담)을 조직도로 설명. 위임 규칙(단일 스킬=직접 / 깊은 작업=부서장 / 다부서 체인=순차·병렬)과 협업 5원칙(피드포워드·단일 보고선·권한 분리·병렬 우선·기록 의무)까지.
3. **사용자 업무 로드** — brand/my-playbook.md(091 산출물)를 읽어 주력 업무·스킬 셋·주간 루틴을 파악한다. 없으면 3문답(주간 반복 업무 / 병목 1개 / 팀 규모)으로 약식 수집하고 091 정식 진단을 권유.
4. **맞춤 팀 설계** — 사용자 업무를 3계층으로 매핑:
   - **주전 부서장 2~4명** — 업무 비중 상위 카테고리의 lead (예: 콘텐츠 중심이면 content-lead+social-lead)
   - **커맨드 1~3종** — 대표 커맨드 15종 중 업무 흐름과 일치하는 체인 (예: 주간 발행 루틴 → /콘텐츠프로덕션)
   - **스케줄 후보 0~2건** — 정기 업무 중 수동 안정화가 끝난 것만 (요일·주기 명시), 발행·발송 게이트는 수동 유지
   - 승인 정책 표(어디서 ⏸ 멈추고 누가 결재하나)를 함께 작성한다.
5. ⏸ **구성안 승인** — 팀 구성안을 제시하고 사용자가 확정/조정한다 (부서장 증감·커맨드 교체·스케줄 보류).
6. **운영 가이드 산출** — 확정안을 주간 운영 루틴(요일별: 무슨 커맨드를 누구에게)·에스컬레이션 규칙(Contract 2회 실패 시 보고, ⛔ 2회 시 사용자 판단)·성장 경로(같은 스킬 5회 → 플레이북 등재, 새 업무 → 092 제작)로 정리하고 my-playbook.md 갱신을 제안한다. 이후 첫 실전 커맨드 1회를 함께 실행해본다.

## Anti-Patterns
- **전 스킬 동시 가동 설계** — 100개를 다 쓰는 팀은 없다. 주전 2~4 부서장으로 시작해 사용 이력으로 넓힌다.
- **가상 아키텍처 해설** — 실제 파일(ROUTING.md·TEAM.md)에 없는 구조를 그림으로 지어내는 것. 설명의 근거는 항상 실물.
- **게이트 없는 자동화** — 스케줄 설계에서 발행·발송 승인 지점을 제거하면 OS가 아니라 폭주 기관차다.
- **진단 없는 처방** — my-playbook.md도 약식 문답도 없이 범용 팀 구성을 내놓는 것. 사용자 업무가 설계의 입력이다.
- **설명만 하고 설계 안 함** — 아키텍처 해설(Phase 1~2)에서 끝내지 않는다. 이 스킬의 산출물은 그 사용자의 팀 구성안이다.

## 활용
- **학습 완주자의 졸업**: 091 진단 → 092 첫 스킬 제작 → 100 팀 설계 (/마케터OS셋업 체인) — 도구 사용자에서 팀 운영자로.
- **1인 마케터의 가상 팀**: 혼자서 부서장 3명(content·ads·analytics-lead)+주간 루틴을 배선하면 "1인 100역" 체제 — 검수는 perspective-reviewer 5관점이 대신 압박한다.
- **팀 도입 설계**: 실제 팀원에게 부서장 카테고리를 배정하고 승인 정책 표를 결재선으로 치환하면, 같은 구조가 그대로 조직 운영 규정이 된다.
