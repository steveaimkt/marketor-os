# 092 나만의 스킬 만들기 (메타 스킬)

100개로 부족한 그 일 — 당신만의 101번째 스킬을 만든다. 에이전트구축 스킬의 **게이트형 문답 5단계** 계승: 한 번에 하나씩 묻고, 답을 받은 뒤 다음으로. 산출 스킬은 SPEC.md 규격을 따라야 하며, quality-checklist C섹션 심사를 통과해야 OS(plugins/)에 편입된다.

## Contract
1. 진행은 게이트형이다 — 각 단계 끝에서 사용자 답을 받기 전에는 다음 단계로 가지 않으며, 사용자 대신 결정하지 않는다.
2. 산출 SKILL.md는 SPEC.md §2 헤더 19필드(id·chains_to 따옴표 필수)와 §3 본문 5섹션(Contract/Phases/Output Format/Anti-Patterns/활용)을 준수한다.
3. 트리거는 기존 스킬 전체와 MECE 검사를 거친다 — ROUTING.md 대조 결과(겹침 후보와 회피 근거)를 명시한다.
4. routing-eval.jsonl 5줄 이상이 함께 생성된다 (intent는 트리거의 패러프레이즈, expected_skill은 새 스킬 ID).
5. **편입 규정**: 새 스킬은 gates/quality-checklist.md C섹션 5항(헤더 완비·재현 가능 STEP·profile.md 참조·gate 판정 정확·실행 검증 1회) + MECE·eval 2항을 통과해야 plugins/ 에 편입된다 — 미통과 스킬은 초안 상태로 보관하고 미달 항목을 알려준다.

## Phases
1. **업무 확정 (범위 게이트)** — 발굴 문답 3개로 대상을 꺼낸다: "매주 여는 파일/시트는?" · "챗봇에 붙여넣었다 결과를 다시 옮기는 일은?" · "마감 전 몰아서 하는 손작업은?" 고른 업무가 전 과정(기획~발행)이면 **가장 자주·가장 아픈 한 스텝**으로 좁힌다 — 작게 끝내야 성공 경험이 생긴다. ⏸ 확정 후 다음.
   - 그릇 판정을 함께: 절차 고정 = 스킬 / 판단·다단계 = 에이전트 체인 / 상시 감시 = 스케줄 결합 — 스킬 그릇이 아니면 그 사실과 대안을 알려준다.
2. **데이터·레퍼런스 수집** — 완성도는 말이 아니라 파일에서 갈린다: 입력 로우데이터 1개(고객·거래처는 A사·B사 마스킹) + 잘된 결과물 2~3개 + 그 공통 구조 1줄. 없으면 결로아 샘플로 진행하되 "실데이터로 교체" 표시. ⏸
3. **구조 설계** — 문답으로 3요소를 확정: **스텝**(5~7개, 각 입력→처리→출력) · **사람 게이트**(발행·발송·확정 직전 최소 1개 — ⏸ 표기) · **판단 룰**(숫자 임계값·금지 표현 목록 — "적당히 좋으면"은 룰이 아니다). ⏸
   - 안전 규칙 기본 3종을 함께 심는다: 읽기 우선(원본 미수정) · 고객·거래처 마스킹 · 발행 전 사람 확정.
4. **자가 검증 (8항목)** — 구축 전 점검: 병목 한 문장 / 레퍼런스 2~3개 / 결과물 공통 구조 / 스텝 5~7(입력·출력) / 게이트 ≥1 / 숫자 룰 / 안전 규칙(읽기 우선·마스킹) / 트리거 초안. 미달 항목은 해당 Phase로 되돌아가 보강. ⏸
5. **SKILL.md 생성 + MECE 검사** — 워크시트 답변을 SPEC 규격으로 조립: 19필드 헤더(id는 임시 "u01" 등 사용자 네임스페이스) + 5섹션 본문. 트리거를 ROUTING.md 100개와 대조해 겹침 후보를 조정하고, routing-eval.jsonl 5줄을 생성한다.
   - 겹침이 크면 신규 제작 대신 기존 스킬 복제·개조 경로를 우선 제안한다 (Phase 3부터 재진입).
6. **시범 실행 + 수정** — 새 스킬을 Phase 2의 실데이터로 1회 완주하고, 기대와 다른 곳을 한 번에 하나씩 고친다 (보통 3~5회). 완주 로그는 편입 심사의 "실행 검증 1회" 증빙이 된다.
7. ⏸ **편입 심사 + 등록 안내** — quality-checklist C섹션 5항 + MECE·eval 2항을 체크리스트로 심사한다. 전 항목 통과 시 plugins/{카테고리}/skills/ 등록 경로와 ROUTING.md 추가 행(1줄)을 안내, 미통과 시 미달 항목과 보강 방법을 반환한다. 완료 후 100(오케스트레이션)으로 핸드오프 — 새 스킬을 내 팀 운영에 배선한다.

```

### ④ 편입 심사 체크리스트 ⏸
```

## Anti-Patterns
- **문답 없이 대필** — 사용자 업무를 안 듣고 일반론 스킬을 지어내는 것. 판단 기준은 업무를 아는 본인만 안다.
- **전 과정 한 방 스킬** — "마케팅 전부 자동화" 스킬은 실패 설계. 범위 게이트에서 한 스텝으로 잘라야 한다.
- **감각 룰** — "톤이 좋으면 통과" 같은 판단 룰 금지. 숫자 임계값·금지어 목록으로 바꿔 적는다.
- **심사 없는 편입 선언** — C섹션 심사를 건너뛰고 plugins/ 에 넣으면 라우팅 테이블이 썩는다. 미통과는 초안 보관이 원칙.
- **게이트 없는 발행 스킬** — 산출물이 발송·발행으로 이어지는데 사람 게이트가 0개면 설계 반려 대상 (SPEC의 gate/mutating 필드도 함께 오기입된다).

## 활용
- **주간 리포트 스킬화**: 매주 손으로 만들던 성과 정리를 5단계 문답으로 스킬화 — 다음 주부터 트리거 한 줄로 호출.
- **팀 공용 스킬 제작**: 팀원 각자의 반복 업무를 이 스킬로 표준화해 편입 심사를 거치면, 팀 전체가 같은 품질 기준의 스킬 셋을 공유한다.
- **커뮤니티 편입**: 편입 심사 통과본은 Certified Builder 표기와 함께 plugins/ 편입 후보가 된다 — 심사 체크리스트가 곧 편입 요건.
