# 066 KPI 트리 설계

> 북극성 지표 1개 → 팀이 직접 움직일 수 있는 **입력 지표**로 분해 → 지표마다 경고/위험 임계값(알림 기준)까지.
> 계승: marketing-os `growth-audit`의 진단 프레임(지표 분해 → 병목 특정 → 실행 액션) + 실습 6-5의 임계값 개념("임계값 = 알람이 울리는 기준선").

## Contract
- 북극성 지표는 정확히 **1개**다 — 후보가 여럿이면 선정 근거를 비교표로 제시하고 사용자가 고른다.
- 트리의 말단(leaf)은 전부 **입력 지표**(팀이 직접 움직일 수 있는 것: 발행 수·발송 수·광고비·전환율)여야 한다 — 결과 지표(매출)가 말단에 남으면 미완성.
- 모든 노드에 ① 계산식 ② 데이터 소스 ③ 측정 주기 ④ 현재값을 붙인다 — 현재값 미상이면 `[측정 필요]` 태그 (숨기지 않는다).
- 임계값은 경고/위험 2단으로 설정하고 산출 근거(목표 대비 % 또는 최근 추세 대비 σ)를 병기한다 — 감으로 찍은 기준선 금지.
- 트리 깊이는 2~3단, 말단 지표는 12개 이하로 제한한다 — 전부 추적하자는 트리는 아무것도 추적하지 않는다.

## Phases
1. **목표·현황 확인** ⏸ — 비즈니스 목표(예: "분기 매출 3억")와 현재 추적 중인 지표·소스를 받는다. [brand/profile.md](../../../../brand/profile.md) §4 숫자·§5 채널 현황 자동 참조 — §4가 미기록이면 임계값을 추세 기준으로만 설계하고 명시한다.
2. **북극성 후보 선정** ⏸ — 후보 2~3개를 "고객 가치 반영 · 매출 선행성 · 측정 가능성" 3기준으로 비교 제시 (예: 구독 비즈니스 = 활성 구독자, 커머스 = 재구매 포함 월 구매자수). 사용자 확정 후 진행.
3. **트리 분해** — 북극성을 곱셈/덧셈 구조로 분해한다. 예: 월 매출 = 트래픽 × 전환율 × AOV → 트래픽 = Σ채널별(오가닉·광고·CRM) → 각 채널의 입력 지표(발행 수·광고비·발송 수)까지. 분해마다 "곱인가 합인가"를 명시해 수식이 검산 가능하게 한다.
4. **지표 정의표** — 노드별 계산식·소스(GA4/광고 관리자/시트)·주기(일/주/월)·담당·현재값. `[측정 필요]` 지표는 확보 방법 1줄을 붙인다.
5. **임계값 설계** — 지표별 2단: 🟡 경고 = 목표 경로의 90% 하회 (또는 최근 12주 평균 -1σ) · 🔴 위험 = 80% 하회 (또는 -2σ). 선행 지표(입력)에 더 민감한 기준을, 후행 지표(매출)에 완충을 준다 — 알람은 원인에서 울려야 대응 시간이 생긴다.
6. **점검 리듬·체인** — 주간 점검 지표(선행) vs 월간 점검 지표(후행)를 나누고 → 062 마케팅 대시보드 빌더로 핸드오프 (이 트리의 지표 정의표가 대시보드 KPI 카드 스펙이 된다).

## Output Format
```
## KPI 트리 — {브랜드} · 목표 "{분기 목표}"
판정 요약: 북극성 = {지표} (현재 {값}) · 말단 입력 지표 {n}개 · [측정 필요] {n}개 · 최우선 병목 후보 {지표}

### ① 북극성 선정 근거 (후보 비교) ⏸
### ② KPI 트리
{북극성}
├─ {지표A} (×) = {계산식}
│   ├─ {입력 지표a1} · ├─ {입력 지표a2}
└─ {지표B} (+) …

### ③ 지표 정의·임계값 표
| 지표 | 계산식 | 소스 | 주기 | 현재값 | 🟡 경고 | 🔴 위험 | 근거 |
|---|---|---|---|---|---|---|---|
| 전환율 | 구매/세션 | GA4 | 주 | 1.8% | <1.6% | <1.4% | 목표 경로 90/80% |

### ④ 점검 리듬 — 주간: {선행 지표들} / 월간: {후행 지표들}
다음 액션: → 062 마케팅 대시보드 빌더 (이 정의표 그대로 KPI 카드 세팅)
```

## Anti-Patterns
- **북극성 복수 선정** — "매출도 팔로워도 중요"는 우선순위 부재 선언이다. 1개 확정, 나머지는 트리 안으로.
- **결과 지표로 끝나는 트리** — 말단이 "매출"이면 팀은 내일 아침 뭘 해야 할지 모른다. 손잡이(입력 지표)까지 내려간다.
- **계산식 없는 노드** — 부모×자식이 검산 안 되는 트리는 그림일 뿐이다. 곱/합 관계를 수식으로 명시.
- **전 지표 동일 임계값** — 모든 지표에 "-10%면 경고"를 일괄 적용 금지. 변동성 큰 지표(일 트래픽)와 안정 지표(AOV)의 기준은 달라야 한다.
- **측정 불가 지표 은폐** — 현재값을 모르는 지표를 아는 척 채우면 트리 전체 신뢰가 무너진다. [측정 필요] + 확보 방법.

## 활용
- **분기 목표 킥오프**: 목표 확정 직후 트리를 만들어 팀별로 입력 지표를 배정 — "매출 3억"이 "주간 콘텐츠 4건·발송 2회·광고비 X"로 번역된다.
- **병목 진단 루틴**: 북극성이 부진할 때 트리를 위에서 아래로 내려가며 어느 가지가 목표 경로를 이탈했는지 특정 — growth-audit식 병목 진단이 상시 가능해진다.
- **알림 자동화 기반**: 임계값 표를 그대로 감시 기준으로 등록하면 (경고/위험 2단) 리포트를 열기 전에 문제가 먼저 알려온다.
- **보고 체계 정렬**: 067 월간 리포트의 경영진 요약을 이 트리 순서(북극성 → 1단 가지)로 구성 — 회의 때마다 지표 정의 논쟁이 사라진다.

> 부록 — 업종별 북극성 분해 시작점 (3단계 후보 제시용)

| 비즈니스 | 북극성 후보 | 1단 분해 |
|---|---|---|
| 커머스 | 월 구매자수 (재구매 포함) | 신규 구매자 + 재구매자 |
| 교육·서비스 | 월 유료 수강/계약 수 | 리드 × 전환율 |
| 콘텐츠·구독 | 활성 구독자 | 신규 유입 − 이탈 |
