# 078 VOC 취합·분류 시스템

> 흩어진 고객의 소리를 채널별 수집 표준 → 4+1 카테고리 분류 → 개선 백로그(020 RICE) 연결까지 상시 파이프라인으로 만든다. 006 리뷰 마이닝이 "일회 심층 분석"이라면 078은 "매주 도는 시스템"이다.
> 계승: marketing-os voc-analyzer의 5축 스키마·긴급도 즉시 플래그 / 072 CS 매뉴얼의 감정점수·에스컬레이션 / 006 리뷰 마이닝의 클러스터링 원칙.

## Contract
1. 모든 VOC 항목은 4+1 카테고리(제품/배송/CS/가격 + 기타) 중 정확히 1개와 감정(-2~+2)·긴급도(0~3)를 받는다 — 기타 비중이 15%를 넘으면 카테고리 추가를 제안한다.
2. 카테고리별 집계에는 원문 인용 근거가 최소 1개씩 붙는다 — 건수만 있는 집계표는 만들지 않는다.
3. 긴급도 3(환불 지연·오배송·안전 이슈)은 주간 집계와 별개로 즉시 플래그해 에스컬레이션 목록으로 분리한다.
4. 반복 상위 이슈는 020 RICE 입력 형식({이슈, 건수, 영향 추정, 원문 인용 3개})으로 변환되어 백로그 연결표에 담긴다 — 분류만 하고 개선으로 안 넘어가는 시스템 금지.
5. 개인 식별 정보는 전 구간 마스킹된다 (이름 첫 글자·연락처 제거).

## Phases
1. **브랜드 컨텍스트 로드** — brand/profile.md(업종·제품·판매 채널) 자동 참조.
2. **채널 인벤토리** — 보유 VOC 소스 확정: 자사몰·입점몰 리뷰 / CS 메일·채팅 / SNS 멘션 / 설문·NPS(070 산출물 재사용). 소스별 접근 방법 명시 — 미연결 채널은 "수동 붙여넣기·CSV 내보내기" 표준으로 대체하고 [수동] 표기.
3. **수집 표준 정의** — 채널별 {수집 주기(기본 주 1회), 형식, 담당}과 정규화 스키마 `{일시, 채널, 원문, 고객ID(마스킹), 별점(있으면)}` 확정 — 출처마다 포맷이 달라도 같은 스키마로 정규화.
4. **분류 태깅** — 항목별: 카테고리(제품/배송/CS/가격/기타) + 감정(-2~+2) + 긴급도(0~3). 긴급도 3은 즉시 플래그 → 에스컬레이션 목록 (072 CS 매뉴얼 기준 연동). 판정이 갈리는 항목은 대표 5건의 분류 근거를 풀어서 기록 (재현성).
5. **집계·추이** — 카테고리별 건수·비중·평균 감정 + 전기 대비 증감. 신규 등장 표현(전기에 없던 불만)은 별도 표기.
6. **개선 백로그 연결** — 반복 상위 이슈(건수 순 + 감정 가중)를 020 RICE 입력 형식으로 정리: 건수가 Reach 근거, 감정×매출 관련성이 Impact 추정 근거가 된다.
7. ⏸ **백로그 이관 확정** — 연결표를 사용자가 검토·승인 → 020 제품 개선 백로그 우선순위로 핸드오프. (심층 원인 분석이 필요한 이슈는 보조로 006 리뷰 마이닝 제안.)

## Output Format
```
## VOC 파이프라인 리포트 — {브랜드} · {기간} · 총 {N}건
판정 요약: 최다 카테고리 {이름}({%}) · 긴급 플래그 {n}건 · 백로그 이관 후보 {n}건

### ① 채널별 수집 현황
| 채널 | 수집 방법 | 주기 | 이번 기간 건수 |
|---|---|---|---|

### ② 분류 집계표
| 카테고리 | 건수 | 비중 | 평균 감정 | 전기 대비 | 대표 인용 |
|---|---|---|---|---|---|
(기타 {%} — 15% 초과 시 카테고리 추가 제안)

### ③ 긴급 에스컬레이션 (긴급도 3 — 즉시 처리)
| # | 채널 | 원문 요약 | 권고 조치 |

### ④ 개선 백로그 연결표 (020 RICE 입력 형식) ⏸
| 이슈 | 건수(Reach 근거) | 영향 추정 | 원문 인용 3개 |
|---|---|---|---|

다음 액션: 승인된 연결표로 020 제품 개선 백로그 우선순위(RICE 스코어링)를 바로 실행할 수 있습니다.
```

## Anti-Patterns
- **일회 분석과 혼동** — "이번 리뷰 500건 깊이 파줘"는 006 담당. 078은 수집 표준·주기가 있는 상시 체계를 만든다.
- **분류하고 방치** — ④ 연결표가 비어 있으면 VOC는 감상문이다. 상위 이슈 수 = RICE 입력 행 수.
- **긴급 건을 주간 리포트에 묻기** — 환불 지연 항의를 다음 주 집계에서 발견하는 구조는 시스템이 아니라 사고 유발기다. 긴급도 3 즉시 플래그.
- **채널 편식** — 리뷰만 모으면 구매자의 소리만 남는다. CS(불만 심층)·SNS(비구매자 인식)를 빼먹지 않는다.
- **카테고리 무한 증식** — 항목마다 새 카테고리를 만들면 추이 비교가 불가능해진다. 4+1 고정, 기타 15% 초과 시에만 신설.

## 활용
- **주간 VOC 루틴**: 매주 월요일 채널별 수집→분류→집계 1회전 — ②표의 전기 대비 열이 쌓이면 그 자체가 품질 추이 대시보드가 된다.
- **개선 회의 안건 자동화**: ④ 연결표를 그대로 020에 넣어 월간 개선 회의 안건을 데이터로 확정 — "목소리 큰 불만"이 아니라 "많고 무거운 불만"부터.
- **CS·제품 연결 고리**: 072 CS 매뉴얼의 에스컬레이션과 긴급 플래그를 연동해, CS가 받은 불만이 제품 백로그까지 흐르는 단일 배관을 만든다.
- **설문 데이터 합류**: 070 설문·NPS 분석의 주관식 응답을 같은 스키마로 정규화해 합류 — 리뷰·CS·SNS·설문 4계통이 한 집계표에 모인다.
- **여정 맵 근거 공급**: 카테고리별 집계·인용을 073 고객 여정 맵핑의 감정·마찰 행 근거로 재사용 — VOC 시스템이 여정 맵의 데이터 소스가 된다.
